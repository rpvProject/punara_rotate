# Punara — Deployment Guide (GitHub → custom GoDaddy domain, HTTPS, auto-deploy)

*Written after inspecting the actual repository. Every recommendation below is
based on what the code actually is, not an assumption. Follow it top to bottom.*

> **What gets deployed:** the marketing site in **`site/`** only. The `lens/`
> platform (FastAPI + a second Next.js app + SQLite/DuckDB) is your internal
> product and stays **local** — it is not part of this deployment. Whenever this
> guide says "the site," it means `site/`.

---

## Recommendation up front

**Host the site on Cloudflare Pages, connected to your GitHub repo, with the
domain's DNS moved to Cloudflare.** Reasoning is in Phase 3. If you do not want
to move nameservers off GoDaddy, use **Netlify** (Phase 4, Path B). A GitHub
Pages fallback is already wired in the repo (Phase 4, Path C).

Why not "just Vercel" (the usual Next.js default): the site is now a **static
export** — it uses none of Vercel's server-side Next.js advantages — and
Vercel's free Hobby tier disallows commercial use, so a consultancy site would
need Pro at \$20/mo. Cloudflare Pages gives equal-or-better performance and
unlimited commercial bandwidth for free.

Already correct in the repo:

- **`site/package.json`** — `build` = `next build`; `dev`/`preview` bind `127.0.0.1`. ✔

- No Dockerfile / docker-compose needed (static site; the repo's root `docker-compose.yml` is for the *future* Lens stack, unrelated).

Generated by this guide:
- **`site/public/_headers`** — security headers (`X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `HSTS`) + a long-cache rule for hashed assets, plus a commented CSP starter. Read by Cloudflare Pages and Netlify; ignored by GitHub Pages.

Optional hardening (not required):
- Pin the build Node version by adding to `site/package.json`: `"engines": { "node": ">=20" }`, and set `NODE_VERSION=20` in the host's build settings.

**One thing you must do before any host can build the current code:** the static
version lives on your **local** branch; GitHub still has the older server
version. Push the current code first (see Phase 6, "Getting the current code to
GitHub"). Cloudflare/Netlify build *from GitHub*, so GitHub must be current.

## Phase 3 — Why Cloudflare Pages (platform comparison)

| Platform | Runs this (static) | Free tier, commercial | CDN / performance | Preview deploys | Custom domain + auto-SSL | Verdict |
|---|---|---|---|---|---|---|
| **Cloudflare Pages** | ✅ | ✅ free, **unlimited bandwidth**, commercial OK | ★★★★★ (330+ edge cities, strong in India + globally) | ✅ | ✅ free | **Recommended** |
| Vercel | ✅ | ⚠️ Hobby is **non-commercial**; Pro \$20/mo | ★★★★★ | ✅ (best-in-class) | ✅ free | Great DX, but you're paying for SSR features a static site doesn't use |
| Netlify | ✅ | ✅ free 100 GB/mo bandwidth, commercial OK | ★★★★ | ✅ | ✅ free | Best "keep GoDaddy DNS" option |
| GitHub Pages | ✅ static only | ✅ free, commercial OK | ★★★ (Fastly) | ❌ no PR previews | ✅ free | Zero-cost fallback; already wired |
| Render / Railway | ✅ (as a static site) | ⚠️ free tiers sleep / limited | ★★★ | partial | ✅ | Built for servers — overkill here |
| AWS (S3 + CloudFront) / Amplify | ✅ | ❌ pay-as-you-go | ★★★★★ | ✅ | ✅ | Enterprise-grade but real ops overhead + cost |

**Decision:** Cloudflare Pages, judged on the criteria you named:
- **Performance** — largest global edge network; consistently fast first-byte in India *and* abroad, matching Punara's India-first, globally-ambitious audience.
- **Cost** — genuinely free for commercial use with **unlimited bandwidth**, so a PR spike (e.g. your benchmark-report launch) has no cost cliff. Vercel meters bandwidth and forbids commercial use on free; Netlify caps at 100 GB/mo.
- **Scalability** — static files on a global CDN scale to any traffic with zero configuration.
- **Maintenance** — Git-connected; every push auto-builds and deploys; nothing to patch or keep running.
- **Security** — Cloudflare's edge adds free DDoS protection and (once DNS is on Cloudflare) WAF/bot controls for the whole domain; a static site is also the smallest possible attack surface.
- **Future growth** — if the site ever needs server logic again, Cloudflare **Pages Functions / Workers** add serverless at the edge without changing hosts.

Runner-up **Vercel** is the pick only if you want the smoothest Next.js DX and will pay Pro for commercial use. **Netlify** is the pick if you refuse to move DNS off GoDaddy (Phase 4, Path B).

## Phase 4 — Domain connection (GoDaddy)

You bought the domain on GoDaddy. GoDaddy stays your **registrar**; the only
question is who runs **DNS**. Pick one path.

> **Which GoDaddy records to DELETE (all paths):**
> In GoDaddy → your domain → **DNS → Manage DNS**:
> - **Delete** the default parked `A` record on host `@` (points at a GoDaddy
>   parking IP — it's what shows the "future home of…" page).
> - **Delete** any `CNAME` on host `www` that points to a GoDaddy parking/`@` target.
> - **Turn OFF Domain Forwarding** (GoDaddy → Domain Settings → Forwarding).
>   Forwarding silently injects its own `A` record that will fight yours.
>
> **Which records to KEEP:**
> - **`MX` records and their `TXT` (SPF/DKIM/DMARC)** *if you use email on this
>   domain* (e.g. hello@your-domain via Google Workspace/GoDaddy email). **Do not
>   delete these** — losing them breaks email. If you have none yet, there's
>   nothing to keep.
> - System `NS`/`SOA` records (managed automatically).
> - Any `TXT` domain-verification records you rely on.

### Path A — Cloudflare Pages + Cloudflare DNS (recommended)

> ⚠️ **DESTRUCTIVE-CHANGE WARNING — moving nameservers.** This path changes your
> domain's nameservers from GoDaddy to Cloudflare, which moves **all** DNS for
> the domain. **Before you switch:** in Cloudflare's DNS screen, confirm every
> existing record (especially `MX`/email and any `TXT`) has been imported/recreated.
> Cloudflare auto-imports most during setup — **verify email records are present**
> or email will stop working. If the domain is brand new with no email, this risk
> is zero.

1. Create a free Cloudflare account → **Add a site** → enter `YOUR_DOMAIN` → Free plan.
2. Cloudflare scans and imports your current DNS records. **Review them** — make sure MX/email records are there.
3. Cloudflare shows **two nameservers** (e.g. `xxx.ns.cloudflare.com`).
4. In GoDaddy → your domain → **Nameservers → Change → Enter my own nameservers** → paste Cloudflare's two → Save. (Propagation: minutes to a few hours.)
5. Once Cloudflare shows the domain "Active," go to Cloudflare **Pages** (Phase 3 setup below) and add the custom domain — apex (`YOUR_DOMAIN`) and `www` are added with one click each (Cloudflare handles apex via CNAME flattening automatically).

### Path B — Netlify, keeping GoDaddy DNS (no nameserver move)

Use this if you don't want to move DNS off GoDaddy. In GoDaddy → **Manage DNS**, after the deletions above, add:

| Type | Host / Name | Value | TTL |
|---|---|---|---|
| `A` | `@` | `75.2.60.5` | 600 |
| `CNAME` | `www` | `YOUR-SITE.netlify.app` (Netlify gives you the exact target) | 600 |

Then in Netlify → **Domain settings → Add custom domain** → enter `YOUR_DOMAIN`; Netlify verifies the records and issues SSL.

### Path C — GitHub Pages, keeping GoDaddy DNS (free, already wired)

The repo has a manual GitHub Pages workflow (`.github/workflows/deploy-site.yml`). In GoDaddy → **Manage DNS**, after the deletions, add:

| Type | Host / Name | Value |
|---|---|---|
| `A` | `@` | `185.199.108.153` |
| `A` | `@` | `185.199.109.153` |
| `A` | `@` | `185.199.110.153` |
| `A` | `@` | `185.199.111.153` |
| `CNAME` | `www` | `rpvproject.github.io` |

Then: repo **Settings → Pages → Source = GitHub Actions**; set repo **Variables** (`SITE_URL`, `BOOKING_URL`, `WEB3FORMS_KEY`, `CUSTOM_DOMAIN=YOUR_DOMAIN`); **Actions tab → "Deploy site to GitHub Pages" → Run workflow**; then **Settings → Pages → Custom domain = YOUR_DOMAIN**.

**Verify DNS (any path):** `nslookup YOUR_DOMAIN` and `nslookup www.YOUR_DOMAIN`, or use dnschecker.org. Wait until it resolves to the host before expecting SSL.

## Phase 3-setup — Cloudflare Pages, click by click (the recommended host)

*(Do this once the repo is on GitHub — Phase 6.)*

1. **Cloudflare dashboard → Workers & Pages → Create → Pages → Connect to Git.**
2. **Authorize GitHub**, select `rpvProject/punara_rotate`.
3. **Build settings:**
   - Production branch: **`main`**
   - Framework preset: **Next.js (Static HTML Export)** — or "None" and set commands manually.
   - **Root directory: `site`** ← critical (monorepo; the site is in `site/`).
   - Build command: **`npm run build`**
   - Build output directory: **`out`**
4. **Environment variables** (Settings → Environment variables → Production): add `NEXT_PUBLIC_SITE_URL=https://YOUR_DOMAIN`, `NEXT_PUBLIC_WEB3FORMS_KEY=<your key>`, `NEXT_PUBLIC_BOOKING_URL=<your calendar>`, and `NODE_VERSION=20`.
5. **Save and Deploy.** First build takes ~1–2 min; you get a `*.pages.dev` URL.
6. **Custom domains** (Pages project → Custom domains → Set up a domain) → add `YOUR_DOMAIN` and `www.YOUR_DOMAIN` (works instantly because DNS is on Cloudflare via Path A). Cloudflare issues SSL automatically.

## Phase 5 — HTTPS

All three hosts issue and **auto-renew** free SSL (Cloudflare Universal SSL / Let's Encrypt). You do not manage certificates.

- **Cloudflare:** SSL is automatic. Turn on **SSL/TLS → Edge Certificates → Always Use HTTPS** (redirects HTTP→HTTPS) and **Automatic HTTPS Rewrites**. Set SSL mode to **Full**.
- **Netlify:** Domain settings → **HTTPS → Verify DNS / Provision certificate**, then **Force HTTPS**.
- **GitHub Pages:** Settings → Pages → tick **Enforce HTTPS** (available once the cert issues, minutes–hours after DNS resolves).

HTTP→HTTPS redirect is automatic on all three once the above is on. The `HSTS` header in `_headers` (Cloudflare/Netlify) tells browsers to always use HTTPS.

## Phase 6 — GitHub automation

**Getting the current code to GitHub (do this first):** your local `main` has the
up-to-date static site *and* the private `blueprint/`. The publish flow keeps the
blueprint out of the public repo by pushing a code-only branch to GitHub's `main`.
Ask me to "sync to GitHub" and I'll push the current code, or if you're doing it
yourself, push a blueprint-free branch to `origin main`. Cloudflare builds
whatever is on GitHub `main`.

*(Alternative that removes the two-branch dance entirely: make the GitHub repo
**private** and push everything — Cloudflare/Netlify/Vercel all build from private
repos via their GitHub app, and the blueprint is then safe because the repo is
private. Your call.)*

Once connected, on Cloudflare Pages (and Netlify/Vercel):
- **Auto-deploy on push:** every push to `main` triggers a build + deploy. No manual step.
- **Preview deployments:** every other branch / PR gets its own temporary `*.pages.dev` URL to review before merging.
- **Rollback:** Pages project → **Deployments** → pick a previous successful deploy → **Rollback**. Instant.
- **Branch strategy:** `main` = production. Work on `feat/*` branches → open a PR → check the preview URL → merge to `main` → production updates. Never edit production by hand.

## Phase 8 — Security

- **Exposed secrets:** none. The Web3Forms key is *public by design* (tied to your inbox, safe in the client). There are **no server secrets** in the site. (Verified: no API keys/tokens in tracked files.)
- **Environment variables:** all `NEXT_PUBLIC_*` and non-sensitive; set on the host, not committed. `.env` is gitignored.
- **Security headers:** shipped via `site/public/_headers` → `out/_headers` (Cloudflare Pages / Netlify). Verify on the live site with securityheaders.com; enable the CSP (commented in the file) once analytics/booking embeds are final.
- **CORS:** N/A — static assets; Web3Forms handles its own CORS.
- **Rate limiting:** Cloudflare provides edge rate-limiting/bot protection; Web3Forms rate-limits submissions server-side; the form's honeypot blocks basic bots.
- **HTTPS:** enforced (Phase 5).
- **robots.txt / sitemap:** both generated and live.
- **Compression:** Brotli/gzip automatic on Cloudflare/Netlify — nothing to configure.
- **Caching:** hashed `/_next/static/*` assets marked `immutable` (1 year) via `_headers`; HTML revalidates.

## Phase 9 — Performance

- **CDN:** automatic and global on Cloudflare Pages — this is the biggest single win, already covered by choosing the host.
- **Image optimization:** static export disables Next's image optimizer (`unoptimized: true`), so **pre-optimize any raster images you add** (WebP/AVIF, right dimensions). Today the site uses SVG/CSS mockups + a generated OG image, so there's nothing heavy to optimize.
- **Compression:** Brotli automatic (Phase 8).
- **Lazy loading:** add `loading="lazy"` to any below-the-fold `<img>` you introduce later.
- **Caching:** configured (Phase 8).
- **Edge functions:** not needed now; available via Cloudflare Pages Functions if the site ever needs server logic.
- **Bundle optimization:** Next already code-splits per route; the site is tiny and static. Re-check bundle size only if you add heavy client libraries.

## Phase 10 — Future improvements

- **Monitoring:** enable **Cloudflare Web Analytics** (free, cookieless, privacy-friendly — no consent banner needed) for traffic; add **UptimeRobot** (free) for an uptime ping on the domain.
- **Error logging:** the static site has little to log; when the **Lens platform** is deployed later, put **Sentry** on it (frontend + API + workers) — see `blueprint/08_technical_architecture.md`.
- **CI/CD improvements:** add a lightweight PR check Action that runs `npm ci && npm run build && npm run lint` in `site/` so a broken build is caught before merge (deploys already run on merge to `main`).
- **Backup strategy:** the repo (Git) is the source-of-truth backup; export contact submissions periodically from the Web3Forms dashboard; when Lens deploys, add scheduled DB backups.
- **Database scaling:** N/A for the site; relevant to Lens (Postgres + ClickHouse path) per the blueprint when that platform is deployed as a separate app.
- **Infrastructure:** keep the marketing site (static, this guide) and the Lens platform (server + DB, deployed separately, later, with your explicit approval) as **two independent deployments** — never couple them.
